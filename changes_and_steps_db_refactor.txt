==============================================================
  SERVER MANAGER — DATABASE SERVER REFACTOR: DETAILED STEPS
==============================================================

OVERVIEW OF CHANGE
------------------
The old "DatabaseServer" model/table has been split into two:

  OLD:
    database_server  (db_id, db_type, user_id, password, schema,
                      version, vm_id FK→virtual_machine, port,
                      description, db_name)

  NEW STRUCTURE:

  1. database_server  (NEW TABLE)
     Columns: db_id, name, focal_point, location, port, remarks,
              vm_id FK→virtual_machine
     Purpose: Represents a logical database server installed on a VM.

  2. schema_info  (RENAMED + ALTERED from old database_server)
     Columns: db_id, db_type, user_id, password, version, port,
              description, db_name,
              database_server_id FK→database_server
     Removed: schema (column dropped), vm_id (column dropped)
     Purpose: Holds per-schema connection credentials, linked to a
              DatabaseServer.

  RELATIONSHIP CHAIN (double join):
    SchemaInfo ──(database_server_id)──► DatabaseServer ──(vm_id)──► VirtualMachine

  A @property `vm` on SchemaInfo resolves the VM transparently:
    schema_instance.vm  →  schema_instance.database_server.vm


==============================================================
DATABASE CHANGES (RAW SQL — run manually in order)
==============================================================

Run these SQL statements against your database BEFORE running
Django migrations:

-- Step 1: Create the new database_server table
CREATE TABLE database_server (
    db_id       INTEGER PRIMARY KEY AUTOINCREMENT,
    name        VARCHAR(100) NOT NULL,
    focal_point VARCHAR(100),
    location    VARCHAR(100),
    port        INTEGER,
    remarks     TEXT,
    vm_id       INTEGER REFERENCES virtual_machine(vm_id) ON DELETE CASCADE
);

-- Step 2: Rename old database_server to schema_info
ALTER TABLE database_server RENAME TO schema_info;
-- NOTE: On MySQL/PostgreSQL, do step 1 first under a different
-- name, then rename:
--   CREATE TABLE database_server_new (...); 
--   ALTER TABLE database_server RENAME TO schema_info;
--   ALTER TABLE database_server_new RENAME TO database_server;

-- Step 3: Add the new FK column to schema_info
ALTER TABLE schema_info ADD COLUMN database_server_id INTEGER
    REFERENCES database_server(db_id) ON DELETE CASCADE;

-- Step 4: Drop removed columns from schema_info
--   SQLite does not support DROP COLUMN before version 3.35.
--   For SQLite < 3.35, recreate the table:
ALTER TABLE schema_info DROP COLUMN schema;
ALTER TABLE schema_info DROP COLUMN vm_id;
--   If on older SQLite, recreate schema_info without those columns.

-- Step 5: Update SoftwareDatabaseMap FK
--   The 'db_id' column in software_database_map already points to
--   the old database_server(db_id). Since we renamed the table to
--   schema_info, the FK now points to schema_info(db_id). No column
--   change needed — just ensure the FK constraint is updated in your
--   DB if you use enforced FK constraints.


==============================================================
DJANGO MIGRATION (0013)
==============================================================

After running the raw SQL above, run:

    python manage.py migrate

Migration 0013 syncs the Django ORM state:
  - Registers new DatabaseServer model (managed=False, db_table='database_server')
  - Registers new SchemaInfo model (managed=False, db_table='schema_info')
  - Updates SoftwareDatabaseMap.db FK to point to SchemaInfo

==============================================================
DOUBLE JOIN QUERY SUMMARY
==============================================================

"Show all schema info for a given VM":
  SchemaInfo.objects.filter(database_server__vm=vm)
  → SQL:
    SELECT schema_info.*
    FROM schema_info
    JOIN database_server ON schema_info.database_server_id = database_server.db_id
    JOIN virtual_machine ON database_server.vm_id = virtual_machine.vm_id
    WHERE virtual_machine.vm_id = <vm_id>

The @property SchemaInfo.vm provides transparent access:
  schema.vm  →  schema.database_server.vm  →  VirtualMachine or None
